## 二、项目背景与市场分析

### 2.1 行业痛点

在企业日常运营中，文档处理占据了大量人力资源：

1. **人工数据录入**：财务报表、人事表格、项目汇报等需要从多个源文档中手动提取数据并填入模板
2. **格式不统一**：Word、Excel、TXT、Markdown 等格式混杂，缺乏统一处理工具
3. **重复劳动**：同一份数据可能需要填入多个不同模板，效率低下且容易出错
4. **公文写作门槛高**：格式规范、用语严谨的公文写作需要专业经验

### 2.2 市场机会

- 全球智能文档处理市场预计 2026 年规模达 $77 亿（CAGR 30%+）
- 国内企业数字化转型加速，AI 办公工具需求激增
- 大语言模型（LLM）技术成熟度快速提升，API 调用成本持续下降

### 2.3 竞品分析

| 竞品 | 优势 | 不足 |
|------|------|------|
| WPS AI | 品牌影响力，生态完善 | 不支持自定义模板填表 |
| 百度文心智能文档 | AI 能力强 | 功能聚焦写作，缺乏填表功能 |
| 传统 RPA 工具 | 自动化成熟 | 基于规则，不理解语义 |
| **DocAI（本项目）** | **AI语义理解 + 自动填表** | **初创阶段，生态待完善** |

## 四、技术方案

### 4.1 技术选型

| 技术 | 选型理由 |
|------|----------|
| Spring Boot 3.5 | Java 生态主流、开箱即用、社区活跃 |
| Vue 3 Composition API | 响应式编程、性能优秀、学习成本低 |
| Element Plus | 企业级 UI 组件库，开箱即用 |
| 智谱 GLM-4-Flash | 免费 API、中文能力强、响应速度快 |
| Apache POI | Office 文档读写行业标准 |
| MyBatis-Plus | 简化数据库操作、自动分页 |
| MySQL | 稳定可靠、运维成本低 |

### 4.2 核心算法 — 自动填表引擎

```
输入：源文档集 D = {d1, d2, ..., dn}，模板文件 T

算法步骤：
1. PARSE: 对每个 di 调用 DocParseService.parseDocument(di) 获取文本 content_i
2. AGGREGATE: 将所有 content_i 拼接/摘要为 combinedText
3. ANALYZE: 调用 DocParseService.analyzeTemplate(T) 获取模板字段 fields[]
4. EXTRACT: 调用 AIService.extractStructuredData(combinedText, fields) 获取 Map<field, value>
5. FILL: 调用 DocParseService.fillTemplate(T, dataMap) 填充模板
6. OUTPUT: 返回填充后的文件 bytes

关键处理：
- 步骤2：如果 combinedText 超过 4000 字符，分段调用 AI 摘要
- 步骤4：AI 返回 JSON 格式，使用 JSON 解析 + 正则表达式双重容错
- 步骤5：Word 模板使用 {{占位符}} 替换，Excel 模板使用空单元格匹配
```

### 4.3 稳定性设计

| 策略 | 实现方式 |
|------|----------|
| AI 重试 | 指数退避（1s → 2s → 4s），最多 3 次 |
| 超时控制 | AI 调用 120s 超时，HTTP 请求 60s |
| 异常处理 | 全局 @ControllerAdvice + 前端拦截器 |
| 大文档 | 自动分段摘要，避免 token 超限 |
| 连接池 | HikariCP 连接池，最大 20 连接 |

| 测试项 | 测试内容 | 预期结果 |
|--------|----------|----------|

| 指标 | 目标值 |
|------|--------|
| 单文档填表时间 | ≤ 90s |
| 页面首屏加载 | ≤ 2s |
| 并发上传 | 支持 5 个文件同时上传 |
| AI 响应时间 | ≤ 30s (非填表场景) |

- 5 个 .docx 文件（每个 ≥ 500KB）

- 3 个 .txt 文件（每个 ≥ 15KB）
